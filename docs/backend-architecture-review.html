1Executive summary

This is a carefully engineered codebase with strong operational instincts: ingestion is idempotent end-to-end, raw provider payloads are stored for replay, OAuth tokens are encrypted with a versioned format, production config is validated at startup, and logging is structured throughout. The provider-sync subsystem (ProviderSyncAdapter + ProviderSyncPipeline) is genuinely good design — ports and adapters done right.

The dominant problem is structural repetition that grows linearly with every new metric. Eleven metric families each carry their own models, write repository, read repository, canonical-derivation repository, derivation service, and query service — and every cross-cutting type (MetricCreatedCounts, MetricWriteResult, DerivedRebuildRequest, the DI modules, the response DTOs) has one hand-written field or branch per family. Adding one metric today touches roughly 15 files. This is the thing to fix before the codebase doubles again.

The second theme is performance debt in the derivation hot path that directly contradicts the repo's own AGENTS.md guidance: timestamps are carried as String in row models and re-parsed (including inside sort comparators), and the canonical bucketing algorithm is quadratic per key. For a single user today this is invisible; on a multi-year backfill it will not be.

Top five actions, in order of leverage:

#ActionWhy
1Introduce a metric-module registry and collapse the per-metric field/branch explosion into Map<MetricKind, …>Cuts ~40% of the code that must be touched per new metric; removes the largest source of future bugs
2Make row models carry Instant, not String; fix the quadratic bucket scanHot-path correctness with your own stated rules; large backfill speedup
3Replace per-route manual auth with Ktor's Authentication plugin and authenticate {} blocksOne forgotten authenticateProtected() call = a silently public endpoint
4Give failed derived rebuilds a repair path (pending-rebuild table + sweeper)Today a failed rebuild leaves canonical tables silently stale until unrelated data arrives
5Batch metric inserts during ingestion; batch rebuild dates per transactionRemoves N+1 round trips on the two write-heavy paths

What's already good (keep it):

  • Idempotent ingestion: batchExternalId + unique indexes + failed-batch external-ID release for retries
  • Raw sourcePayload + normalized payload stored per batch — replay/audit is possible by design
  • ProviderSyncPipeline with adapter ports: token refresh, throttling, caching, partial failure handling all provider-agnostic
  • algorithmVersion on all canonical tables — derivations are versioned and re-runnable
  • Startup config validation that actively rejects placeholder/local values in production (AppConfig.kt)
  • AES-GCM token encryption with a v1: format prefix (rotation-ready), constant-time API-key comparison
  • Derived rebuilds moved out of the ingestion transaction (recent commit) — the right call
  • Testcontainers-backed integration tests split from fast unit tests; OpenAPI exported as a build artifact

2Architecture as it stands

api/ Routes.kt (907) · OpenApi.kt (1138) · Errors.kt · Auth.kt · dto/ ↓ calls application/ IngestionService → IngestionMappingService (1095) → MetricWriteService DerivedRebuildExecutor → 9 × Canonical*DerivationService metric/<family>/ {QueryService, derived/, repository/} ← Exposed DSL lives here providersync/ {Pipeline, Ports, Adapters} Scheduled sync · Sync jobs · Provider workflow/status/catalog ↓ infrastructure/ database/ (tables, Hikari, Flyway) · repositories/ (support, oauth, jobs) security/ (ApiKeyHasher, TokenCipher) · config/ · logging/ domain/ HealthRecord (sealed) · MetricTypes · HealthProvider ← imports application.* external/ google/ · withings/ (clients, normalizers, providers, sync adapters)

Data flow: trusted scripts or provider adapters POST normalized batches → batch + records + raw payload persisted in one transaction → per-record metric writes with duplicate skipping → affected UTC dates collected → after commit, canonical/derived tables are rebuilt per date (delete-and-reinsert, keyed by algorithmVersion). Reads go almost exclusively through the canonical tables.

3Architecture findings

A1 · Adding a metric requires shotgun surgery across ~15 files Critical

application/metric/** · MetricWriteService.kt · DerivedRebuildExecutor.kt:20–42 · IngestionService.kt:160–246 · di/AppModules.kt · domain/IngestionModels.kt

Every metric family is hand-instantiated everywhere it appears. The pattern is plainly visible in the "affected dates" plumbing — nine parallel linkedSetOf<LocalDate>() accumulators in IngestionService.ingestBatch, nine parallel fields on MetricWriteResult, nine on DerivedRebuildRequest, nine if (… .isNotEmpty()) blocks in TransactionalDerivedRebuildExecutor, twelve fields on MetricCreatedCounts and its DTO twin, and a 12-arm when in MetricWriteService.write. The same expansion repeats in the DI modules, the catalog service, and the OpenAPI descriptors.

This is the single biggest long-term cost in the codebase: each new metric multiplies boilerplate, and each cross-cutting change (e.g. "track affected dates differently") must be edited eleven times, with the compiler only partially able to help.

Fix: introduce a MetricKind enum and a MetricModule interface that bundles what a family contributes: writer, derivation, catalogDescriptor, and readService. Register modules in a list (mirroring the existing — and well-done — HealthDayModuleRegistry and HealthProviderRegistry patterns). Then: 
data class MetricWriteResult(
    val created: Map<MetricKind, Int> = emptyMap(),
    val duplicateSkipped: Int = 0,
    val affectedDates: Map<MetricKind, Set<LocalDate>> = emptyMap(),
)

class TransactionalDerivedRebuildExecutor(val modules: List<MetricModule>) {
    override suspend fun rebuild(request: DerivedRebuildRequest, at: Instant) =
        modules.forEach { m ->
            request.affectedDates[m.kind]?.takeIf { it.isNotEmpty() }?.let { dates ->
                suspendTransaction(db) { m.derivation.recompute(dates, at) }
            }
        }
}
Adding a metric becomes: one module class + one table + one migration + one registry entry. MetricCreatedCountsResponse can stay a typed DTO for API stability — build it from the map at the boundary.

A2 · The layering is inverted in three places — either enforce it or simplify it High

domain/HealthProvider.kt:3 · application/IngestionService.kt:3–6 · application/metric/*/repository/*

The package names promise hexagonal layering (domain ← application ← infrastructure/api), but the dependencies don't hold:

  • domain/HealthProvider.kt imports application.providersync.ProviderSyncProgressSink — the innermost layer depends outward.
  • IngestionService (application) consumes and produces api.dto types (IngestionBatchRequest, IngestionSummaryResponse) — the use-case layer is welded to the HTTP contract.
  • Repositories full of Exposed DSL live under application/metric/*/repository, while other repositories live in infrastructure/repositories. Same role, two homes, and "application" now hard-depends on Exposed.

Half-enforced layering is worse than none: it misleads readers about what is safe to import where.

Fix: pick one of two honest options. (a) Enforce it: move ProviderSyncProgressSink into domain (it's a pure interface), have ingestion accept a domain command object mapped from the DTO at the route layer, and move all Exposed repositories under infrastructure. (b) Accept that a single-user service doesn't need four layers and reorganize by feature (ingestion/, metrics/steps/, providers/withings/…), each feature owning its routes, service, and repository. Given the codebase already groups metric code by family, (b) is the smaller step and reads better. Either way, add a lightweight architecture test (e.g. Konsist) so the chosen rule is mechanical, not aspirational.

A3 · Routes.kt (907 lines) and the 17-field service bag Medium

api/Routes.kt · api/Application.kt:98–156 · api/OpenApi.kt (1138 lines)

Every endpoint, its handler, and its inline OpenAPI .describe block live in one file, and every handler receives the full ApplicationServices bag (17 services + the database) regardless of what it needs. The bag is assembled by hand from Koin in buildApplicationServices() — 17 inject lines mirrored by 17 constructor arguments mirrored by 17 fields.

Fix: split routing into per-resource extension functions — Route.ingestionRoutes(ingestion: IngestionService), Route.providerRoutes(…), Route.metricReadRoutes(…), Route.adminRoutes(…) — each taking only its own dependencies (resolved by inject() inside each module, which deletes the bag entirely). Keep the .describe DSL but move shared descriptor builders next to the routes they document.

A4 · Two dependency-injection idioms fight each other Medium

di/AppModules.kt:139 · MetricWriteService.kt:29–37 · HeartRateQueryService.kt:18 · all *WriteRepository

Koin is the declared container, but many classes also default-construct their own dependencies: MetricWriteService() news up all eight write repositories (which are not registered in Koin at all), and query services default-create canonical repositories (= CanonicalHeartRateDerivationRepository()) that are registered as singletons. The result is two object graphs — the Koin one and the shadow one created by default args — and it is luck, not design, which instance a given code path uses. Because the repositories are stateless this is currently harmless, but it makes substitution in tests inconsistent and will bite the first time a repository gains state (a cache, a metric counter).

Fix: delete every default-argument instantiation; require constructor injection and register everything in the modules. As a rule: = SomeClass() in a constructor signature is allowed only in test fixtures.

A5 · A job-queue abstraction that nothing uses Low

application/metric/common/MetricDerivedBuilder.kt · CanonicalHeartRateDerivation.kt:44–57

DerivationJobInput carries attemptCount, lastError, lockedAt, and reason — the vocabulary of a persistent job queue — but every implementation hard-codes them (attemptCount = 0, lastError = null, lockedAt = null). Every derivation also constructs a Clock.fixed(computedAt, UTC) just to thread a timestamp through the builder. This is speculative generality: the machinery of a queue with none of its behavior.

Fix: either delete the unused fields and pass computedAt directly (smallest change), or actually build the queue — which finding C1 argues you want anyway. Don't keep the husk.

A6 · Three clock abstractions Low

infrastructure/time/UtcClock.kt · ProviderSyncPipeline.kt:18 (() -> Instant defaulting to Instant.now()) · derivation builders (java.time.Clock)

Services use UtcClock, the sync pipeline takes a () -> Instant lambda whose default bypasses UtcClock entirely, and the derivation layer uses java.time.Clock. Time is the classic source of flaky tests; one seam is enough.

Fix: standardize on java.time.Clock (it's the JDK seam, Clock.fixed ships with it) or on UtcClock everywhere, injected — never defaulted to a real clock inside the class.

4Performance findings

P1 · Row models carry timestamps as String and re-parse them in hot loops High

HeartRateModels.kt:6 (and every sibling *Models.kt) · CanonicalHeartRateDerivation.kt:106,126 · CanonicalTimestampBuckets.kt:28 · PostgresTypes.kt:13

The database hands back OffsetDateTime; repositories immediately stringify it (toApiString()) into row models like HeartRateSampleRow(measuredAt: String); the derivation layer then parses it back with Instant.parse — inside a sort comparator:

// CanonicalTimestampBuckets.kt — measuredAt(it) is { Instant.parse(it.measuredAt) }
val timedRows = rows
    .sortedWith(compareBy<T> { measuredAt(it) })  // parses O(n log n) times
    .map { row -> CanonicalTimedRow(row, groupKey(row), measuredAt(row)) } // parses again
…
    .sortedBy(measuredAt)  // and again on the way out

A day with 10,000 heart-rate samples performs on the order of 150,000+ Instant.parse calls per rebuild — and rebuilds run per affected date, per metric, on every ingest. This is exactly the "repeated expensive conversions" your AGENTS.md forbids. Strings exist in these models only because the same row type is reused as a quasi-DTO.

Fix: repository row models keep Instant (the DB already gives you the typed value — row[…].toInstant()); convert to ISO strings only in MetricResponseMappers at the DTO boundary. In canonicalTimestampRows, map to CanonicalTimedRow first, then sort by the precomputed field. This is mechanical and removes parsing from the hot path entirely.

P2 · Quadratic bucket scan in canonical timestamp dedup High

CanonicalTimestampBuckets.kt:41–53

For each row, the algorithm linearly scans all existing buckets for its key (bucketsForKey.firstOrNull { abs(…) <= 30s }). With one source and one context — the common case — a dense day degenerates to O(n²): 10k samples ≈ 50M duration comparisons. The input is already sorted by time, which makes the scan unnecessary.

Fix: since rows arrive time-ordered, only the last bucket per key can possibly be within tolerance (any earlier bucket's representative is even further in the past). Replace the scan with a check against bucketsForKey.lastOrNull() — O(n) total. (This also subtly changes semantics from "first matching bucket" to "most recent bucket", which is the more defensible clustering anyway; note it in the algorithm version if you care about byte-stable output.)

P3 · Ingestion writes metrics one row, one statement at a time Medium

IngestionService.kt:173–192 · MetricWriteService.kt · all *WriteRepository

The ingest loop dispatches each record to a per-record insertIgnoreAndGetId. A Withings backfill batch with 5,000 samples issues 5,000 sequential INSERT round trips inside one transaction (plus the overlap SELECTs of P4). The duplicate-detection already lives in unique indexes, so nothing requires row-at-a-time execution.

Fix: group records by type up front (records.groupBy { it::class }), then have each writer accept a list and use Exposed's batchInsert(ignore = true) (ON CONFLICT DO NOTHING). Created-vs-skipped counts come from the returned row count. The "affected dates" computation is pure and stays as is.

P4 · Per-record overlap query for Google step samples, with a redundant in-memory recheck Medium

StepWriteRepository.kt:41–59

Every Google step interval triggers a SELECT scanning candidate rows, then re-verifies overlap in Kotlin — but the SQL predicate (startAt < candidateEnd AND endAt > candidateStart, with the ±1s widening) already implies the Kotlin condition, so rows are materialized just to confirm what the WHERE clause proved. It's also an N+1 against the ingest batch.

Fix: short term, replace the fetch-and-check with an EXISTS/empty() probe and drop the ±1s widening so SQL and intent match exactly. Long term, you already have a GiST index on the time range (step_samples_source_instance_time_range_gist_idx) — promote the rule to a Postgres exclusion constraint (EXCLUDE USING gist (source_instance_id WITH =, tstzrange(start_at, end_at) WITH &&) scoped to that source) and let inserts fail-fast instead of pre-querying.

P5 · Derived rebuild: one transaction per metric group, one job per date, all sequential Medium

DerivedRebuildExecutor.kt:57–132 · CanonicalHeartRateDerivation.kt:27–33

After a backfill touching 90 days, the executor runs up to 9 metric groups × 90 dates of delete-and-reinsert work as ~9 sequential transactions whose bodies loop dates one at a time (and recompute itself loops dates.forEach { builder(...).processJob(...) }, constructing a fresh builder per date). Each date's input load is a separate full-day SELECT.

Fix: three cheap wins, in order: (1) pass the whole date set into one delete (date inList dates) + one batched insert per metric; (2) reuse one builder across dates; (3) since the rebuild already runs post-commit, run independent metric groups concurrently with a small Semaphore-bounded coroutineScope. Pairs naturally with the queue in C1.

P6 · Blocking JDBC runs on whatever dispatcher calls it Medium

BaseReadService.kt:19–22 · IngestionService.kt:62 · all suspendTransaction(db = database) call sites

suspendTransaction is called without a coroutine context, so the blocking JDBC work executes on the caller's dispatcher — Ktor's request worker pool for HTTP paths, Dispatchers.Default for the scheduler. A handful of slow queries (a long backfill read, a lock wait) can occupy CPU-pool threads and stall unrelated request handling. With maxPoolSize connections, there is also no backpressure alignment between coroutines admitted and connections available.

Fix: define one shared dispatcher sized to the pool — Dispatchers.IO.limitedParallelism(config.maxPoolSize) — and pass it at the single wrapper: suspendTransaction(dbDispatcher, db = database) { … }. Route every transaction through that wrapper (you already have dbQuery; ingestion and the executors should use the same one).

P7 · Source-instance filter resolved by an extra query on every read Low

BaseMetricRepository.kt:51–68

Any read filtered by provider/providerInstanceId first runs a separate SELECT to resolve matching source-instance IDs, then injects them as inList. That's one extra round trip per request for data that is tiny and nearly immutable (you create a source instance once per provider account).

Fix: cache the (provider, providerInstanceId) → sourceInstanceId map in memory with invalidation on resolveOrCreateSourceInstanceInTransaction — or just inline the join into each query. The cache is ~20 lines and removes the round trip everywhere at once.

5Correctness & robustness

C1 · A failed derived rebuild leaves canonical tables silently stale High

IngestionService.kt:254–301 · IngestionRepository.markDerivedRebuildFailed

Moving the rebuild out of the ingestion transaction was correct, but the failure path only marks the batch (derived_rebuild_failed) and logs. Nothing retries: raw metric rows are committed, canonical tables for those dates keep yesterday's contents, and every read endpoint (which reads canonical) serves stale data until some future ingest happens to touch the same dates. The same applies to the partial-failure design inside the transaction: a mid-batch exception commits the already-written metric rows (reasonable — retries dedupe), but those rows never get a rebuild either.

Fix: persist the work. Write a pending_derived_rebuilds(metric_kind, date, attempt_count, last_error, locked_at) row set in the ingestion transaction itself (it's tiny), have the executor delete rows it completes, and add a sweeper to the existing scheduler loop that retries leftovers with backoff. This is precisely the shape DerivationJobInput already models (A5) — build it for real. Bonus: an admin endpoint "rebuild date range" becomes trivial and gives you a manual repair tool.

C2 · Retryability decided by substring-matching error messages Medium

ScheduledProviderSyncService.kt:206–211
val nonRetryable = errors.any { error ->
    error.contains("validation", ignoreCase = true) ||
    error.contains("not configured", ignoreCase = true) || …

The pipeline goes to real effort to produce structured ProviderSyncError(dataType, code, message) — and then this code flattens errors to strings (joinToString) and greps the prose. Reword one message ("needs reconnect" → "requires re-authentication") and a permanent failure becomes an infinite retry loop.

Fix: keep ProviderSyncError objects through executeConfig and classify on code (e.g. a NonRetryableCodes set, or better: a retryable: Boolean the adapter sets, since the adapter is the only party that knows). Flatten to a string only for the persisted lastErrorMessage.

C3 · Lifecycle states are stringly-typed everywhere Medium

"processed" / "failed" / "partial_failed" in IngestionService, ProviderSyncPipeline:199–203, repositories, DTOs · account statuses in ProviderOAuthRepository

Batch status, sync status, job status, and account status are raw strings compared with == at every decision point (existingBatch?.status == "processed"). A typo compiles; an enum wouldn't. The set of states and their transitions is real domain logic currently living in nobody's head but the grep index.

Fix: one enum per lifecycle (BatchStatus, SyncStatus, AccountStatus) with fromDb/dbValue mapping at the repository boundary. Serialize the same wire strings so the API is unchanged.

C4 · Single-process assumptions are documented — keep them deliberate Info

ScheduledProviderSyncService.kt:21 · ProviderSyncJobService (in-memory CoroutineScope)

The run guard is an in-process mutex (the comment says so honestly), and background sync jobs execute on an in-memory scope with interrupted-job marking at startup — good. This is all fine for a single-instance deployment; just be aware that the moment a second replica appears, you need a DB-backed claim (SELECT … FOR UPDATE SKIP LOCKED on due configs/jobs) rather than the mutex. The schema already supports it; only the guard changes.

C5 · Scheduled sync re-fetches the full lookback window every run Low

ScheduledProviderSyncService.kt:230–241

syncWindow starts at checkpoint − lookbackDays, so with the default 7-day lookback and daily cadence, each run re-downloads ~7 days that are already ingested. Idempotent batch caching absorbs most of the cost, but the provider API calls still happen, and the batch-external-ID granularity means a window shifted by hours produces new batch IDs (cache misses) every run.

Fix: separate the concerns: sync forward from checkpoint normally, and run the lookback re-scan on a slower cadence (e.g. once daily) or align windows to day boundaries so the batch cache actually hits.

6Security & API surface

S1 · Auth is opt-in per route — one missed line publishes an endpoint High

Routes.kt:890–898 · every handler's first line

Authentication is a manual call.authenticateProtected(services) call at the top of each of ~40 handlers. Nothing structural prevents the next endpoint from omitting it — and the codebase already shows how routine drift happens (S2). For a health-data service, "secure by default" should be a property of the routing tree, not of developer discipline.

Fix: install Ktor's Authentication plugin with a bearer provider that wraps the existing SupportRepository/ApiKeyHasher lookup, then group routes: 
routing {
    healthRoutes()                          // public: /admin/health, oauth callback
    authenticate("api-key") {              // everything else, by construction
        ingestionRoutes(); providerRoutes(); metricReadRoutes(); adminRoutes()
    }
}
This composes with the per-resource split in A3 and deletes ~40 boilerplate lines. While you're there, consider a tiny RateLimit install on the auth failure path; SHA-256 for high-entropy generated keys is fine, but unlimited guessing is free today.

S2 · Newer endpoints are missing OpenAPI documentation Low

Routes.kt:735–775 (blood-pressure, cardiovascular, extended-body-measurements)

Six read endpoints have no .describe block, so they appear in the generated contract only via code inference (if at all) — while every older endpoint is documented. Classic drift, and evidence for moving docs closer to route modules (A3) where a missing description is conspicuous.

Fix: add the six .describeReadOperation(…) calls; optionally add a test that walks the routing tree and asserts every non-hidden route has an operationId — you already have the OpenAPI export test to hang it on.

S3 · List endpoints cannot paginate — nextCursor is hardwired null Medium

ReadDtos.kt:19 · MetricCatalogService ("no cursor for now") · QueryParams.limit

Reads accept limit (cap 1000 presumably) with no offset or cursor; the response meta even declares a nextCursor field that is always null. A client wanting a full year of heart-rate samples must shard by time ranges manually and guess whether a window was truncated.

Fix: keyset pagination is nearly free given the existing ordering: cursor = base64 of (measuredAt, id) of the last row, decoded into a (col > ts) OR (col = ts AND id > lastId) predicate in BaseMetricRepository. The DTO field already exists, so this isn't even a breaking change.

7Readability cleanups

R1 · IngestionMappingService: 1,095 lines of hand-unrolled validation Medium

application/IngestionMappingService.kt

Eleven mapXxx functions repeat the same five moves: parse instants, check start<end, range-check numbers, collect issues, rebuild the success condition by hand (if (startAt != null && endAt != null && steps > 0 && startAt.isBefore(endAt)) — restating checks already performed above, a divergence bug waiting to happen). The trailing provider!!/ingestedAt!! after the issues check works but reads as a leap of faith.

Fix: a ~50-line validation helper pays for itself eleven times: a FieldValidator accumulating issues with combinators like instant(name), orderedRange(start, end), positive(name, value), and a final buildIfClean { … } so the success condition is "no issues for this record" rather than a re-derived boolean. Also consider serializing normalizedRecordJson once per DTO instead of inside each mapper.

R2 · Zero-filled response boilerplate Low

IngestionService.kt:84–121 · ScheduledProviderSyncService.toDto mappers

The duplicate-batch response spells out twelve zeroed count fields and nine empty date sets. Defaults on the DTO (stepSamples: Int = 0) or a MetricCreatedCountsResponse.ZERO constant collapse ~40 lines to three — and the A1 map refactor deletes the problem class entirely. Similarly, the hand-written toDto() mappers in the scheduled-sync service duplicate mappers that exist elsewhere; one shared mapping file per DTO family avoids drift.

R3 · GoogleHealthConfig and WithingsConfig are the same class twice Low

AppConfig.kt:56–74, 187–205

Identical seven fields, identical validation functions duplicated per provider. One OAuthProviderConfig + a Map<String, OAuthProviderConfig> keyed by provider code makes the third provider a config entry instead of a new data class, new validator, and new AppConfig field.

R4 · MetricsQueryService is 109 lines of pure delegation Low

application/metric/common/MetricsQueryService.kt

Every method forwards verbatim to a per-metric service. After the route split (A3), each route module can inject its per-metric service directly and the facade can be deleted — one less hop for readers tracing a request.

8If I redesigned it — breaking changes I'd make

Would I rewrite this application? No. The bones are good: the ingestion contract, the provider pipeline, and the schema are all worth keeping. But if I were starting from the same requirements — a personal, single-user health hub — I would make five decisions differently, and four of them are reachable from here as deliberate breaking changes.

8.1 · Name the actual architecture: an event log with projections

The system already is event-sourced in disguise: ingestion_batches + raw payloads are an append-only event log; metric tables are a projection; canonical tables are a projection of the projection. But the code treats projections as precious mutable state to be patched per ingest, which is why C1 (drift) and the per-date rebuild plumbing exist. I would make replay a first-class operation: every projection rebuildable from raw batches for any date range with one admin call. Once replay is cheap and trusted, rebuild failures stop being scary (re-run it), the algorithmVersion bump workflow becomes "replay everything at v2", and the pending-rebuild queue (C1) is just the incremental special case of replay. Most of the code for this exists; what's missing is the unifying operation and the confidence to lean on it.

8.2 · Question eager canonicalization itself

Eight canonical tables exist to answer one question at read time: "when two sources report the same moment, which sample wins?" For a single user, the raw tables hold maybe a few million rows — small for Postgres. I would benchmark a read-time alternative before maintaining eight materializations forever: bucket timestamps in SQL (date_bin('30 seconds', measured_at) as a first approximation of the tolerance rule) and pick winners with DISTINCT ON over a provider-rank join. If that holds p95 under target on a year-range query — likely with the indexes already in place — the entire derivation layer for timestamp metrics (~25 files, the rebuild executor, the drift failure mode) disappears, and "canonical" becomes a view definition you can change without rebuilding history. Keep materialization only where the computation is genuinely heavy (sleep-night assembly, step day-bucket contributions). This is the single highest-leverage simplification available, and it's breaking only internally — the API shape stays.

8.3 · One generic samples table per value shape, not one table per metric family

Greenfield, I would not create heart_rate_samples, respiratory_rate_samples, hrv_samples, cardiovascular_measurements, extended_body_measurements… — they are all (source_instance_id, provider_record_id, measured_at, metric_type, value, unit, context) with minor variations; three of them literally share that exact shape already. I'd keep structurally distinct shapes separate (step intervals, sleep sessions+stages, daily activity summaries) and fold the point-in-time scalars into one scalar_samples table partitioned by metric type if needed. That single change is what makes A1's metric-module registry natural instead of forced: a new scalar metric becomes a row in a catalog, not a migration + table + five repositories. From here, this is a real migration with real risk; I'd only do it riding on the 8.2 simplification, and it's defensible to skip — but it's what I'd do differently from scratch.

8.4 · Decide single-user out loud (and what multi-user would cost)

There is no user_id anywhere — correct for the stated scope, and adding speculative tenancy now would be waste. But make it an explicit recorded decision (an ADR in docs/): if this ever serves two people, every metric table, every canonical table, every unique index, the API-key model (keys ↔ user), and the OAuth account model change. The cheap insurance available today is keeping source_instance_id as the universal scoping column it already is — a future user_id can hang off source_instances alone, which is exactly one table away from everything.

8.5 · API v2 hygiene, batched for one deliberate break

When a v2 is warranted, bundle the contract-level fixes in one go rather than dribbling breaks: enum-typed status fields (C3); cursor pagination as the default list shape (S3); metricsCreated as a map keyed by metric kind instead of a twelve-field object (so new metrics stop being API-breaking at all); and ISO-8601 fields formally typed as format: date-time in the schema. Everything else about the API — resource naming, error envelope with requestId and field details, discovery endpoints — I would keep exactly as is; it's a well-designed surface.

What I would explicitly keep

  • Ktor + Exposed + Flyway + Koin — right-sized; a heavier framework buys nothing here.
  • The ingestion contract — normalized batches + raw payload + idempotency keys is the best part of the design.
  • The provider adapter pipeline — adding provider #3 is already cheap; don't touch it.
  • Dedicated typed tables for steps, sleep, and activity — their shapes genuinely differ.
  • The operational posture — config validation, structured logs, Prometheus, request IDs, integration-test split.
